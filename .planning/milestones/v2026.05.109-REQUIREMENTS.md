# Requirements: Clara Gateway Consolidation

**Defined:** 2026-01-27
**Core Value:** Single daemon, multiple providers

## v1 Requirements

Requirements for Gateway Consolidation milestone. Each maps to roadmap phases.

### Gateway Architecture

- [x] **GATE-01**: Gateway daemon runs all providers from single process
- [x] **GATE-02**: Gateway supports provider lifecycle management (start, stop, restart)
- [x] **GATE-03**: Gateway provides unified message routing to all providers
- [x] **GATE-04**: Gateway includes protocol versioning for adapter compatibility
- [x] **GATE-05**: Provider architecture supports adding Slack/Telegram later

### Discord Provider

- [x] **DISC-01**: Discord provider integrated into gateway (not separate process)
- [x] **DISC-02**: Discord bot responds to messages with streaming LLM responses
- [x] **DISC-03**: Discord provider maintains message queuing and batching behavior
- [x] **DISC-04**: Discord provider supports multi-model tier selection (!high, !mid, !low)
- [x] **DISC-05**: Discord provider supports image/vision capabilities
- [x] **DISC-06**: Discord provider maintains reply chain tracking
- [x] **DISC-07**: discord_bot.py wrapped by DiscordProvider (REVISED from "deleted")

### Email Provider

- [x] **EMAL-01**: Email provider integrated into gateway
- [x] **EMAL-02**: Email monitoring with rule-based alerts continues working
- [x] **EMAL-03**: Email provider uses gateway event system for alerts
- [x] **EMAL-04**: email_monitor.py deleted completely after migration

### CLI Client

- [x] **CLI-01**: CLI client connects to gateway via WebSocket
- [x] **CLI-02**: CLI messages flow through gateway processor with tool support
- [x] **CLI-03**: CLI client supports both local and remote gateway connections

### Entry Point

- [x] **ENTR-01**: `python -m gateway` is the only entry point needed
- [x] **ENTR-02**: Gateway starts all providers (Discord, Email, CLI) from single command
- [x] **ENTR-03**: Docker Compose runs single gateway service (not multiple)

### Data Preservation

- [x] **DATA-01**: mem0 databases remain untouched and functional
- [x] **DATA-02**: Session history and message storage continues working
- [x] **DATA-03**: Project/user isolation maintained through consolidation

### Feature Parity

- [x] **FEAT-01**: Memory system (mem0) provides context from past conversations
- [x] **FEAT-02**: MCP plugins extend Clara with external tools
- [x] **FEAT-03**: Code execution via Docker/Incus sandbox works
- [x] **FEAT-04**: Hooks and scheduler trigger on events
- [x] **FEAT-05**: Multi-model tier support continues working
- [x] **FEAT-06**: All behavioral tests pass (message dedup, emotional context, etc.)

### Production Readiness

- [x] **PROD-01**: Provider crash triggers auto-restart (not gateway crash)
- [x] **PROD-02**: Rate limiting prevents spam (configurable per provider)
- [x] **PROD-03**: Health check endpoint reports gateway and provider status
- [x] **PROD-04**: Structured logging includes provider context
- [x] **PROD-05**: Gateway handles 100+ concurrent users without degradation
- [x] **PROD-06**: Graceful shutdown completes pending responses

## Out of Scope

Explicitly excluded from Gateway Consolidation milestone. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Changes to mem0 storage | Databases must remain untouched per constraints |
| Slack provider implementation | Architecture only, actual implementation deferred to v2 |
| Telegram provider implementation | Architecture only, actual implementation deferred to v2 |
| Web UI client | Gateway supports it, but building UI is separate work |
| Active-mode batching optimization | Complex feature for 100+ user channels, defer to v2 |
| APScheduler migration | Current scheduler sufficient, defer until scaling needs |
| OpenTelemetry tracing | Basic structured logging sufficient for MVP |
| Circuit breaker patterns | Add when scaling issues appear in production |
| Proactive messages (ORS) | Requires separate planning phase |
